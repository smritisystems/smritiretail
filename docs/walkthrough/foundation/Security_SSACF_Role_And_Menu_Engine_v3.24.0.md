# SSACF — Security & Access Control Framework v3.24.0

## 1. Purpose

Implement the SMRITI Security & Access Control Framework (SSACF): a fully decoupled, policy-based authorization engine with explicit Deny-wins precedence, hierarchical role inheritance, and permission-driven Dynamic Menu resolution. This replaces the legacy hardcoded `UserRole` enum-based access checks for all new modules.

## 5. Architecture Decisions

### Decoupled Policy Layer
```
User → SMRITIUserRole → SMRITIRole (hierarchical) → SMRITIRolePolicy → SMRITIPolicy → SMRITIPolicyPermission → SMRITIPermission
```
Roles do **not** directly own permissions. Policies act as the intermediary. This allows the same policy to be assigned to multiple roles and the same permission to be in multiple policies.

### Explicit Deny Wins (Tri-State Logic)
```
Explicit DENY  →  Wins over all ALLOW (even inherited)
Explicit ALLOW →  Granted unless overridden by DENY
Default        →  DENY (no permission = no access)
```
This is enforced in `SecurityService.resolve_user_permissions()`.

### Role Inheritance
Roles have a `parent_role_id` FK referencing themselves. `get_effective_roles()` walks the tree using a BFS traversal, collecting all ancestor roles. All policies from all inherited roles are compiled.

### Permission Caching
Resolved permissions per user are stored in a module-level dict `_permissions_cache: Dict[str, Set[str]]`. Cache is invalidated on role/policy changes by calling `invalidate_user_permission_cache(user_id)` or `clear_all_permissions_cache()`.

### Dynamic Menu Resolver
`SMRITIMenu` rows carry a `permission` field (permission code). The resolver calls `get_visible_menus()`, which fetches all active menus and filters them by checking if the user has that permission code. SYSADMINs bypass all checks.

## 6. Design Rationale

- **Strangler-fig migration**: SSACF tables are additive. The legacy `UserRole` enum on `User` is **not removed** — it continues to work for all existing logic. SSACF is a new parallel system that new code uses via `require_permission()`.
- **SYSADMIN bypass**: Users with legacy `UserRole.SYSADMIN` automatically pass all `verify_user_permission()` checks to maintain full backward compatibility.
- **Module-level in-memory cache** was chosen over Redis at this stage to avoid adding a new infrastructure dependency. Redis can be introduced when the permission resolution becomes a hot path.

## 7. Implementation Summary

### Phase A: Data Layer
- Created 8 SQLAlchemy models inheriting from `BaseEntity` (auto-UUID, tenant-scoped, soft-delete)
- Created 2 Alembic migration files — the autogenerated file missed the `CREATE TABLE` statements because the table names weren't in the `include_object` whitelist at autogenerate time; a second explicit manual migration was written and applied

### Phase B: Service Layer
- `SecurityService` (async, session-injected) implements role traversal, permission resolution, menu filtering, and audit logging
- Module-level permission cache with explicit invalidation API

### Phase C: API & Guards
- `/api/v1/security/menus` — returns dynamically filtered menus per current user
- `/api/v1/security/check` — runtime permission check for a specific code
- `/api/v1/security/permissions`, `/roles`, `/policies` — admin lookup endpoints, guarded by `SECURITY.VIEW_SETTINGS`
- `require_permission(code)` dependency factory added to `deps.py` for use on any new route

### Phase D: Seed Data
- 13 master lookup types (department, designation, gender, country, state, city, blood_group, gst_category, tax_class, currency, unit, payment_mode, reason_code)
- Default department and designation values
- 5 system roles (SYSADMIN, MANAGER, CASHIER, REPORT_USER, VIEWER)
- 5 permissions (SALES.CREATE, SALES.APPROVE, REPORT.VIEW, ITEM.CREATE, ITEM.DELETE)
- 4 dynamic menus
- 3 default users (super, manager, cashier) with dynamic SSACF role mappings

## 10. Known Limitations

- **Permission cache is in-memory**: Cache is not distributed. In multi-worker deployments, invalidation only clears the cache in the worker that called it. A Redis-backed cache is recommended before horizontal scaling.
- **Menu tree is flat-fetched**: Nested parent/child menu trees are returned as flat lists; frontend must assemble the hierarchy from `parent_id`.
- **No policy UI yet**: SSACF policies, roles, and menus can only be managed via API; no admin UI component exists in this phase.
- **Legacy `UserRole` enum coexists**: The enum is not removed. Until the frontend fully migrates to SSACF, both systems are active simultaneously.
